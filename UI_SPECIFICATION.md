# KinamonKB UI Functional Specification (v1.0)

このドキュメントでは、KinamonKB 承認管理画面の各ボタンの機能とその裏側で発生する処理について詳しく説明します。

---

## 1. News Feed (一次情報プール)
RSSから収集された直後の、AIによる簡易判定（ABC）ステージです。

| ボタン / UI要素 | 機能説明 | 処理の裏側 |
| :--- | :--- | :--- |
| **Fetch RSS** | 登録済みのRSSフィードから最新記事を取得します。 | `fetch_news.py` を実行。記事を収集し、AIがA/B/C判定を行って **News Feed（手動確認エリア）** に溜めます。 |
| **Manage Feeds** | RSS取得元のURL一覧を管理するパネルを開閉します。 | `rss_feeds.json` を読み書きします。 |
| **Run Patrol** | 全自動パイプラインを開始します。 | `auto_patrol.sh` を実行。RSS収集後、**A判定のものは自動で2択案生成まで進め（Pendingへ）、C判定は自動削除**します。 |
| **Manual Add** | RSSにない特定のURLや記事を直接追加します。 | `/bot_01_observer/_news_candidates` に手動MDを作成します。 |
| **A (Adopt)** | その記事を「採用」としてマークします。 | ファイル内の `Evaluation` を `A` に書き換えます。 |
| **B (Hold)** | その記事を「保留（プールに残す）」としてマークします。 | ファイル内の `Evaluation` を `B` に書き換えます。 |
| **C (Delete)** | その記事を「削除」としてマークします。 | ファイル内の `Evaluation` を `C` に書き換えます。 |
| **Apply Decisions** | 選択したA/B/Cの判断を一括で実行します。 | **A**: `generate_opinion.sh` を回し、2択案を生成して `_pending` へ移動。<br>**C**: ファイルを物理削除。<br>**B**: 保存してプールに維持。 |

---

## 2. Pending Tasks (承認待ち・2択決定)
AI（Kina Fox）が生成した2択案から一つを選び、最終的な投稿内容を確定するステージです。

| ボタン / UI要素 | 機能説明 | 処理の裏側 |
| :--- | :--- | :--- |
| **PLAN A / PLAN B** | AIが提示した2つの投稿案のどちらかを選択します。 | 選択した案が下のテキストエリア（Final Post）に反映されます。 |
| **Include Source Link** | 投稿に元記事のURLを含めるかどうかを切り替えます。 | テキストエリアの内容をリアルタイムに再構成します。 |
| **Force Unique (Time)** | 投稿末尾に時刻IDを付与します。 | Xの「重複投稿エラー」を回避するための機能です。 |
| **Advice to Kina Fox** | 判断の理由やボットへのアドバイスを記入します。 | **重要**: この内容は `feedback_knowledge.md` に蓄積され、次回以降のAIの判断に影響を与えます。 |
| **Final Post Area** | 実際に投稿される文章を直接編集できます。 | ユーザーが最終的な微調整（語尾の修正など）を行う場所です。 |
| **Decide & Queue** | 決定した内容を「投稿待ち」状態にします。 | `_pending` から `_post_pool` へ移動し、`growth_log.md` に選択を記録します。 |
| **Hold Task** | 今すぐ判断できないタスクを「保留」一覧に移します。 | フォルダを `_pending` → `_held` へ移動させます。 |
| **Trash** | そのタスクをゴミ箱へ移動します。 | フォルダを `_trash` へ移動させます（復元可能です）。 |

---

## 3. Ready to Post (投稿実行待ち)
承認が完了し、実際に X (Twitter) へ送信する直前のステージです。

| ボタン / UI要素 | 機能説明 | 処理の裏側 |
| :--- | :--- | :--- |
| **Post to X** | 承認済みの内容を実際に X へポストします。 | X APIを叩き、成功すればファイルを `_decided` (履歴) へ移動します。 |

---

## 4. History / Archive (履歴・ゴミ箱)
過去の行動の参照と、ゴミ箱の管理を行います。

| ボタン / UI要素 | 機能説明 | 処理の裏側 |
| :--- | :--- | :--- |
| **View on X ↗** | 実際にポストされた投稿をブラウザで確認します。 | 保存されている `tweet_id` を元に X のURLを開きます。 |
| **Restore** | ゴミ箱にあるアイテムを元の場所（Pending/Heldなど）に戻します。 | メタデータを読み取り、元のディレクトリへファイルを移動します。 |
| **Delete Permanently** | ゴミ箱内のアイテムを完全に削除します。 | ファイルを物理削除します（復元不可）。 |

---

## 5. identity (ボットの成長管理)
日々の承認から学習したAIの「自己更新」を管理します。

| ボタン / UI要素 | 機能説明 | 処理の裏側 |
| :--- | :--- | :--- |
| **Analyze Identity** | 近近の「Decide」の傾向をAIが分析し、提案を作成します。 | `identity-system.md` を使い、`persona.md` の更新案を生成します。 |
| **Approve / Reject** | AIが提示した個々の更新項目を承認/棄却します。 | 適用前のみ操作可能で、適用後は変更不可（ロック）となります。 |
| **Apply to persona.md** | 承認した項目をボットの定義ファイルに正式反映します。 | `persona.md` に追記し、提案を「適用済み」として保存します。 |

---

## 6. System Health (システム状態)
サーバーの負荷や、各フォルダに溜まっているファイル件数を可視化します。

| ボタン / UI要素 | 機能説明 | 処理の裏側 |
| :--- | :--- | :--- |
| **Refresh** | 最新のメトリクス（CPU/メモリ/ファイル数）を取得します。 | サーバーのOS情報と `ls` によるファイルカウントを再実行します。 |
