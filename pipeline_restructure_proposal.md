# 提案: パイプライン再構成 — Step 3 の分解と段階的RAG化

## 現状の問題

### Step 3 が1回のAIコールに詰め込んでいる情報量

| 情報ソース | バイト数 | 推定トークン |
|---|---|---|
| `shared_knowledge.md` | 687 | ~200 |
| `shared_persona.md` | 929 | ~300 |
| `filter-system.md` | 2,681 | ~800 |
| Bot Hobbies（5体分を動的生成） | ~500 | ~150 |
| 記事本文（英語、最大5000文字） | 5,000 | ~1,500 |
| ユーザープロンプト | ~300 | ~100 |
| **合計（入力だけで）** | **~10,097** | **~3,050** |

ローカルAI（n_ctx=4096）では、入力だけで3,050トークンを消費し、モデルの応答用に残るのは**わずか1,000トークン程度**。これにシステムトークン（KVキャッシュ等）のオーバーヘッドが加わり、溢れて落ちる。

### Step 5（2択案生成）はさらに深刻

| 情報ソース | 推定トークン |
|---|---|
| `shared_knowledge.md` + `shared_persona.md` | ~500 |
| ボット固有 `persona.md` | ~200 |
| ボット固有 `tone_and_voice.md` | ~600 |
| `opinion-system-template.md` | ~600 |
| 記事本文（全文） | ~1,500+ |
| ユーザープロンプト | ~200 |
| **合計** | **~3,600+** |

こちらも同様にギリギリで、ペルソナが成長するほどに確実に溢れる構造。

---

## 提案: 「1回1仕事」への分解

### 設計思想
> **「AIには1回のコールで1つの判断だけをさせる」**
> 複数の判断を同時にさせると、コンテキスト消費が倍加するだけでなく、
> モデルの注意力（Attention）が分散して各判断の精度も落ちる。

---

### 新フロー全体図

```
[Step 1] RSS収集・本文抽出       ← 従来通り（プログラム処理）
    ↓
[Step 2] タイトル翻訳            ← 従来通り [🤖 AI]
    ↓
[Step 3a] トピック分類           ← NEW [🤖 AI・軽量]
    ↓
[Step 3b] ボット割当             ← NEW [⚙️ コード処理・AI不要]
    ↓
[Step 3c] 重要度判定             ← NEW [🤖 AI・焦点絞り]
    ↓
[Step 4] 候補保存・UI表示        ← 従来通り（プログラム処理）
    ↓ ← kinamon承認ゲート（人間判断）
    ↓
[Step 5a] 記事の構造化要約       ← NEW [🤖 AI]
    ↓
[Step 5b] 2択案生成              ← 改善 [🤖 AI・要約ベース]
```

### Step 3a: トピック分類 [🤖 AI・軽量]

**目的:** 記事が「何の話題か」だけを判定する。ボットの事も評価の事も一切考えない。

| 項目 | 内容 |
|---|---|
| **入力** | タイトル（日本語）+ 本文冒頭300文字のみ |
| **システムプロンプト** | トピック分類の定義リストだけ（~300トークン） |
| **出力** | `{"topics": ["crypto", "security"], "summary_jp": "一行要約"}` |
| **推定コンテキスト** | **~600トークン（現在の1/5）** |

```
トピック候補（例）:
- ai_llm: LLM・生成AI・プロンプト技術
- ai_tools: AI搭載ツール・開発支援
- crypto_defi: DeFi・DEX・プロトコル
- crypto_regulation: 規制・法律・コンプライアンス
- security: サイバーセキュリティ・ハッキング
- dev_tools: 開発環境・エディタ・OS
- investment: 投資・マーケット分析
- creator_tools: 音楽生成・動画生成・クリエイティブAI
- hardware: ハードウェア・量子コンピュータ・データセンター
- web_culture: web文化・SNS・メディア論
```

> [!TIP]
> このステップは「タイトル＋冒頭のみ」で済むため、4096トークンのローカルAIでも余裕で処理可能。

---

### Step 3b: ボット割当 [⚙️ コード処理・AI不要]

**目的:** トピックタグとボットの趣味（hobby）をマッチングする。

**ここにAIは不要。** 純粋なコードロジック（キーワードマッチング）で十分。

```python
# 例: マッチングロジック（概念）
BOT_TOPIC_MAP = {
    "bot_01_observer": ["dev_tools", "ai_tools"],
    "bot_02_trader":   ["crypto_defi", "crypto_regulation", "investment"],
    "bot_03_creator":  ["creator_tools", "ai_llm"],
    "bot_04_auditor":  ["security", "ai_llm", "hardware"],
    "bot_05_wolf":     ["web_culture", "crypto_defi"],
}

def match_bot(topics: list[str]) -> str:
    scores = {}
    for bot_id, interests in BOT_TOPIC_MAP.items():
        scores[bot_id] = len(set(topics) & set(interests))
    best = max(scores, key=scores.get)
    return best if scores[best] > 0 else "bot_01_observer"
```

> [!IMPORTANT]
> **利点:** AIの気分（ハルシネーション）でボット名を間違える事故が完全になくなる。
> マッチングロジックの定義は `attributes.json` の `hobby` から自動生成も可能。

### Step 3c: 重要度判定 [🤖 AI・焦点絞り]

**目的:** 3aで分類済み、3bで割当済みの記事に対し「読む価値があるか？」だけを判断する。

| 項目 | 内容 |
|---|---|
| **入力** | タイトル + 本文（冒頭1500文字） |
| **システムプロンプト** | `filter-system.md` の3点チェック基準のみ（Bot hobbyリスト不要） |
| **出力** | `{"evaluation": "A", "reason": "理由"}` |
| **推定コンテキスト** | **~1,500トークン（現在の1/2）** |

> [!NOTE]
> `assigned_bot` はもう3bで確定しているため、ここでは問わない。
> `filter-system.md` からもBot Hobby関連の記述を削除でき、プロンプトがさらにスリム化する。

### Step 5a: 記事の構造化要約 [🤖 AI] — RAGへの布石

**目的:** 生の英語記事を、2択案生成に必要十分な「構造化ブリーフ」に圧縮する。

| 項目 | 内容 |
|---|---|
| **入力** | 英語記事の本文（全文） |
| **システムプロンプト** | 「以下の記事を構造化して要約せよ」（~200トークン） |
| **出力** | 構造化ブリーフ（~300トークン） |

```markdown
## Structured Brief
- **What:** Circle社のUSDCインフラでDriftプロトコルが$285Mハック被害
- **Who:** Circle, Drift Protocol, SUI Blockchain
- **Impact:** DeFi信頼性への懸念、ステーブルコイン依存リスクの顕在化
- **Key Tension:** インフラ提供者（Circle）の責任範囲 vs プロトコル自身のセキュリティ
```

> [!TIP]
> **RAGへの発展:**
> この構造化ブリーフを保存・ベクトル化しておけば、将来的に
> 「過去に似たテーマの記事でどんな意見を出したか」を検索して
> コンテキストに注入できる。これがRAGの第一歩になる。

### Step 5b: 2択案生成 [🤖 AI・改善版]

**目的:** 構造化ブリーフ（生記事ではない）をベースに、ボットの個性で2択案を作る。

| 項目 | 内容 |
|---|---|
| **入力** | 構造化ブリーフ（~300トークン）← 生記事の代わり |
| **システムプロンプト** | ボット固有の `persona.md` + `opinion-system-template.md` |
| **出力** | 2択マークダウン |
| **推定コンテキスト** | **~1,800トークン（現在の1/2）** |

> [!IMPORTANT]
> `shared_knowledge.md` と `shared_persona.md` はここでは不要。
> ボットの個性（persona）と出力フォーマット（template）だけで十分。
> ペルソナが将来大きく成長しても、RAGで関連部分だけ取り出せば制御可能。

## 将来のRAG化ロードマップ

### Phase 1（今回の提案）: パイプライン分解
- 各ステップを独立させ、1回1仕事の原則を確立
- 構造化ブリーフの保存を開始

### Phase 2: ベクトルストアの導入
- 構造化ブリーフと過去の判断結果をベクトルDB（例: ChromaDB）に保存
- 新しい記事が来た時、類似記事の過去判断を取得

### Phase 3: 動的コンテキスト注入
- ボットのペルソナが大きくなったら、全文ではなく関連セクションだけを検索・注入
- `growth_log.md` や `feedback_knowledge.md` から関連エントリだけを選択的に取得
- 「このボットは過去にこういう記事でこう判断した」という文脈付きの2択案生成

```
[将来の Step 5b イメージ]
   構造化ブリーフ (300 tokens)
 + ペルソナの関連部分のみ (RAG検索, 200 tokens)
 + 類似記事の過去判断 (RAG検索, 200 tokens)
 + 出力テンプレート (300 tokens)
 ─────────────────────────────
   合計: ~1,000 tokens（余裕）
```
